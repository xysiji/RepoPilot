# RepoPilot 项目规格

## 1. 项目定位

RepoPilot 是一个面向本地 Python Git 仓库的代码分析与修复 Agent。它接收自然语言任务和目标工作区，读取代码、制定计划、生成候选 Patch，在任何真实文件修改之前暂停并等待人工审批；获批后应用修改、执行 pytest、审查 Git Diff，并输出可追溯的最终报告。

项目设置两条清晰的验收线：P0–P5 是可在一个工作日内跑通的教学型核心 Demo，形成一次完整、安全的修复闭环；P6 增加 SQLite 恢复、上下文治理和 Trace 后，才称为 RepoPilot 第一版。P7 的 reviewer subgraph 只是选做增强。三者都面向单用户、单进程场景，不是通用生产级 Agent 平台。

## 2. 目标用户与场景

目标用户是希望分析或修复 Python 项目的本地开发者。第一版支持三类任务：

- 代码分析：定位实现、解释调用关系、给出证据和建议。
- 缺陷修复：根据任务和现有测试生成小范围 Patch，经审批后修改文件。
- 测试修复：读取 pytest 失败信息，重新规划并在有限次数内提出后续 Patch。

典型输入：

- `workspace_path`：已存在的本地 Python Git 仓库路径。
- `user_task`：明确、可验证的分析或修复目标。
- 可选执行约束：最大重试次数、允许的测试目标。

第一版只接受位于配置的工作区根目录之下、工作树干净、文本文件为 UTF-8、可由 pytest 验证的小型项目。不接收压缩包上传，也不克隆远程仓库。

## 3. 第一版功能范围

第一版必须形成以下闭环：

1. 校验工作区、Git 状态和任务参数。
2. 枚举目录并按需搜索、读取相关代码。
3. 由 LLM 生成 Pydantic 校验后的结构化计划。
4. 由只读执行子图收集代码证据并形成结构化修改提案。
5. 用确定性 Python 逻辑生成候选 unified diff 与 Patch 摘要。
6. 通过 LangGraph interrupt 暂停，向用户展示 Patch 哈希、文件列表和 Diff。
7. 仅在用户批准且 Patch/工作区指纹仍匹配时应用修改。
8. 用固定参数的 pytest 子进程执行测试，采集退出码和限长输出。
9. 测试失败时，在重试预算内把失败证据送回规划节点；每个新 Patch 都重新审批。
10. 读取累计 Git Diff，执行确定性审查，生成最终报告。
11. 用 SQLite Checkpointer 保存可恢复的图状态；LangSmith Trace 为显式选择的可选能力。
12. 通过 FastAPI 提供创建 run、查询 run、提交审批三个最小 HTTP 接口。

## 4. 非目标

第一版明确不实现：

- 常驻 daemon、自定义 TCP Socket、NDJSON、JSON-RPC 或多客户端协议。
- Textual TUI、复杂 Web 前端、WebSocket/SSE 实时界面。
- 自定义 EventBus、EventWriter 或自定义 MCP 协议客户端。
- 通用 Bash、任意命令执行、模型可直接调用的通用文件写入工具。
- 向量数据库、消息队列、Kubernetes、分布式任务调度。
- 多 Agent 并行执行、后台子 Agent 轮询、递归 Agent。
- Dify/Coze 编排、数据库业务模型、账号和权限系统。
- 自动提交、推送、建分支或创建 Pull Request。
- 跨语言仓库、二进制文件修改、大规模重构和生产级沙箱。

MCP、Reviewer Subagent、Skills 和复杂上下文压缩只在后续阶段评估，不进入一天 Demo 的关键路径。

## 5. 验收标准

第一版通过以下验收才算完成：

- 给定干净的本地 Python Git 仓库和明确任务，系统能生成结构化计划与候选 Patch。
- 审批前，目标仓库字节级无变化；拒绝审批后，仓库仍无变化。
- 审批请求包含目标文件、unified diff、Patch 哈希、风险说明和 thread ID。
- 批准时会复核工作区边界、文件 preimage 哈希和 Patch 哈希；任一不匹配则拒绝应用。
- `apply_patch` 只能修改提案中列出的、工作区内的文本文件，且不会调用 shell。
- pytest 只能通过固定的 `python -m pytest` 入口运行，参数来自受限 Pydantic 模型，具备超时和输出上限。
- 测试失败最多重试 `max_retries` 次；不存在无限图循环；每次新副作用都需要新审批。
- 最终报告包含原始目标、计划摘要、修改文件、审批记录、测试结果、Git Diff 审查、重试次数和未解决问题。
- 单元测试覆盖节点路由、路径边界、symlink、审批恢复、Patch 哈希、超时和重试上限；集成测试覆盖一次批准路径和一次拒绝路径。
- SQLite checkpoint 能用同一 `thread_id` 从审批中断处恢复。

## 6. 一天核心 Demo 场景（P0–P5）

在 `demo_workspace/` 准备一个小型、干净的 Python Git 仓库，其中有一个可由单元测试复现的简单缺陷。演示流程：

1. 调用 `POST /api/v1/runs`，提交工作区路径和“修复函数边界条件并通过测试”的任务。
2. 查询返回的结构化计划、代码证据和候选 Diff，确认工作区尚未变化。
3. 调用审批接口提交 `approve + patch_hash`，恢复同一个 LangGraph thread。
4. 观察 Patch 被应用、pytest 被执行；若预置第一次方案失败，则展示一次有限重规划和第二次审批。
5. 查询最终状态，读取累计 Git Diff 审查与最终报告。
6. 运行拒绝分支，证明拒绝后不修改任何文件。

Demo 的重点是安全边界和可解释状态变化，不追求 UI、并行度或复杂任务覆盖率。

该 Demo 可以使用 P4/P5 阶段的内存 checkpointer，只证明同一进程内的 interrupt/resume；进程重启后的恢复能力属于 P6 第一版验收，不能用核心 Demo 的结果冒充。

## 7. 已知限制

- 只面向单机、单用户、本地工作区；FastAPI 请求可在 LLM 或 pytest 执行期间保持较长时间。
- 第一版要求工作树初始干净，避免把用户既有改动和 Agent 改动混在一起。
- Patch 仅支持受限文本替换/新建，限制文件数和总字节；不处理二进制文件、重命名、chmod 或 Git submodule。
- pytest 失败不一定由当前 Patch 引起；第一版不自动建立完整基线测试归因。
- LLM 计划和修改提案可能不正确，Pydantic 只能校验结构，不能证明语义正确。
- 多文件应用只能做到“全部预检 + 失败时尽力回滚”，不等同于文件系统事务。
- SQLite 适合本地 Demo，不作为多进程、高并发持久化方案。
- LangSmith 若启用可能上传代码或工具载荷，因此默认关闭，启用前必须配置脱敏与用户同意。
- 代码执行仍发生在宿主机子进程中；固定 pytest 命令降低风险，但不等同于容器沙箱。

## 8. 成功边界

RepoPilot 第一版成功的定义不是“模型能写代码”，而是：模型只负责需要推理的计划和提案，Python 代码负责边界、路由、审批、校验和副作用；任何不确定或失配状态都默认停止，而不是继续写文件。
